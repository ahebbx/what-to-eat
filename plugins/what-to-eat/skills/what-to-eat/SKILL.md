# 今天吃什么

帮用户终结每天"吃什么"的纠结。核心思路：先收集当天的实际情境，再给出**有具体理由的明确推荐**，最后记录下来——记录越多，推荐越准。

## 工作流程

### 1. 判断是哪一餐
用 bash `date` 拿当前时间（注意用户时区）。5:00–10:00→早餐，10:00–15:00→午餐，之后→晚餐（21点后可考虑夜宵）。用户明确说了哪餐就按用户说的。

### 2. 读取饮食记录（本skill的记忆）
记录固定存于 **`~/.what-to-eat/饮食记录.md`**（家目录统一路径——不管在哪个工具、哪个项目里问"吃什么"，读写的都是同一份记录）。
- **找到了**：通读一遍。重点：用户城市、偏好区、最近5–7天吃了什么。
- **没找到**：第一次使用。在提问环节顺便问一下城市和最近两三天吃了什么（用户说不记得也没关系），然后 `mkdir -p ~/.what-to-eat` 并创建文件（格式见下）。
- 若发现工作目录或其他位置有旧的 `饮食记录.md`（早期版本的存放位置），把内容合并进 `~/.what-to-eat/饮食记录.md` 后删除旧文件。
- 无文件系统的环境（如 claude.ai 网页版）：跳过读写，直接靠对话内信息推荐。

### 3. 查天气
用 web search 搜「{城市} 今天天气」。拿到气温和天气状况即可。查不到就跳过，按当前季节推断。

### 4. 问一个问题就够
如果用户消息里没说明，用 AskUserQuestion 问**一次**（不要连环追问）：今天想怎么吃？典型选项：自己做 / 点外卖 / 出去吃 / 都行你定。可在同一次提问里附带问状态（想吃清淡的 / 想吃硬的 / 累了想省事 / 想犒劳自己）。用户消息里已经说清楚的信息绝不重复问。测试或非交互环境下无法提问时，按合理默认值继续。

### 5. 生成推荐
综合以下因素（这是本skill的核心价值——推荐必须是"今天的"推荐，不是万能答案）：

- **天气与时令**：读 `references/seasonal.md`。下雨降温→热汤热面；闷热→凉面凉皮、清爽小炒；应季食材优先。
- **避免重复**：最近3天吃过的菜/品类不推。看到记录里连续出现同类（比如三天面食），主动换赛道。
- **营养均衡**：回看近几天：蔬菜少了补蔬菜，连吃几天外卖建议做顿饭，红肉多了换鱼虾豆制品，重口连着吃推荐清淡的。
- **心情与预算**：累→快手菜（≤20分钟）或省心外卖；想犒劳→硬菜或馆子；月底吃土→经济实惠款。
- **菜品灵感**：读 `references/dishes.md`，从对应分类里挑，也可以推荐库外的菜。

### 6. 输出格式
给 **1个首选 + 2个备选**。首选要果断——用户来找你就是因为选择困难，别甩回十个选项让人再选一遍。

每个推荐包含：
- 吃什么：具体到菜名或店铺品类（"番茄牛腩面"，而不是"中餐"）
- 怎么吃：做饭 / 外卖 / 外食
- 推荐理由：一两句，**必须扣住今天的实际情况**（天气、最近吃过什么、营养缺口、用户状态至少其一）
- 若是做饭：主要食材清单 + 3–5步简要做法；顺手提示需要买什么

理由的好坏标准："好吃又有营养"是废话；"连续三天外卖了，这个番茄豆腐汤15分钟就能做好，正好补补这几天缺的蔬菜"才算合格。

### 7. 记录
用户选定后（或对话收尾时），把这一餐追加进 `~/.what-to-eat/饮食记录.md`。用户如果透露了偏好（"我不吃香菜""最近在控碳水"），同步更新偏好区——这是推荐越来越准的关键。

## 饮食记录.md 格式

```markdown
# 饮食记录

## 基本信息
- 城市: 北京
- 偏好: 不吃香菜；最近想多吃蔬菜

## 记录
- 2026-07-08 午餐: 番茄牛腩面（外卖）
- 2026-07-08 早餐: 包子 + 豆浆（楼下买）
- 2026-07-07 晚餐: 清炒西兰花 + 煎三文鱼（自己做）
```

记录按时间倒序，一行一餐，括号注明方式。

## 原则
- 果断、具体、贴合当天。每次的推荐理由都应该不一样——如果把日期遮住理由也成立，说明写得太模板化了。
- 用户否掉首选就立刻换下一个方向，不解释不辩护。
- 尊重记录里的偏好和忌口，这比任何通用营养建议优先级都高。
